MODULE 5

Tool Schema Anatomy

Building with Claude — Tool Use and Function Integration

Why this matters: A tool call is really three small, ordered contracts — a schema you write, a tool_use block Claude emits, and a tool_result your code sends back. Get the schema's wording wrong and the wrong tool gets called, or the right tool with guessed arguments. Everything below is fully offline and interactive — nothing sends a real request.

3
Vague vs. specific tool descriptions

    
Two tools can share a database — what tells Claude which one to call is the
      wording of description, not the schema's field types. Click
      Simulate on each set and see which tool gets picked for the same request.


    
Request: "I don't have the invoice ID, only that the vendor is
      'Metro Legal' — find and validate their most recent invoice."


    

      

        
❌ Vague descriptions

        
get_invoice_details:
  "Gets invoice info."

search_invoices:
  "Finds invoices."

        
        

          🚨 Wrong tool selected. Claude calls
          get_invoice_details(invoice_id="Metro Legal") — with neither description
          stating what kind of identifier each tool expects, "Metro Legal" gets passed to the tool
          built for an exact ID. The lookup fails, and the agent has to recover mid-conversation
          instead of choosing correctly the first time.
        

      


      

        
✅ Specific descriptions

        
get_invoice_details:
  "... by its exact invoice ID,
   e.g. 'INV-2026-0101'."

search_invoices:
  "... use this when you have
   a vendor name but not an
   invoice ID."

        
        

          ✅ Correct tool selected. "I don't have the invoice ID, only ... the
          vendor" matches search_invoices's description almost verbatim. Claude calls
          search_invoices(vendor_name="Metro Legal") first, then chains
          get_invoice_details on the returned ID — see this exact sequence in
          Lab 5's sixth request.
