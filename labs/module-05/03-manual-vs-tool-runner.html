Manual Loop vs. Tool Runner

Building with Claude — Tool Use and Function Integration

Why this matters: day3/invoice_tool_agent.py and invoice_tool_agent_beta.py solve the exact same task and reach the exact same decisions — the beta tool_runner() doesn't change WHAT happens, only who writes the loop and how a failure is expressed. Everything below is fully offline — nothing sends a real request.

Same failure, two styles

Both agents are asked to look up a vendor ID that doesn't exist (VEND-404). Click Simulate on each to see how that failure is expressed in code — and confirm both still end up as the same wire-format tool_result.

Manual — invoice_tool_agent.py
def get_vendor_details(vendor_id):
    vendor = VENDOR_DB.get(vendor_id)
    if vendor is None:
        return {"error": f"No vendor found "
                          f"with ID '{vendor_id}'"}
    ...

# in the loop:
result = func(**block.input)
tool_results.append({
    "type": "tool_result",
    "tool_use_id": block.id,
    "content": json.dumps(result),
    "is_error": "error" in result,
})
✅ get_vendor_details("VEND-404") returns {"error": "No vendor found with ID 'VEND-404'"}. The loop itself checks "error" in result and sets is_error: true — you write this check yourself, every time.
Beta — invoice_tool_agent_beta.py
@beta_tool
def get_vendor_details(vendor_id: str) -> str:
    """Fetch one vendor's record...

    Args:
        vendor_id: Exact vendor ID.
    """
    vendor = VENDOR_DB.get(vendor_id)
    if vendor is None:
        raise ToolError(
            f"No vendor found with ID "
            f"'{vendor_id}'"
        )
    return json.dumps({...})

# no loop code — the runner catches
# ToolError and builds tool_result itself
✅ raise ToolError("No vendor found with ID 'VEND-404'") is caught by tool_runner(), which builds the identical {"type": "tool_result", ..., "is_error": true} message for you — same wire format, no manual check written.

A real bug this guide warns about

Every beta tool in this lab ends with return json.dumps({...}) — a deliberate choice, not decoration. Click below to see what actually happens if you return the raw dict instead (this is the real error text hit while building this lab).

@beta_tool
def calculate_tds(amount_inr: float, category: str) -> dict:   # ← "-> dict", returning the dict directly
    ...
    return {
        "tds_rate_pct": rate * 100,
        "tds_amount_inr": tds_amount,
        "net_payable_inr": round(amount_inr - tds_amount, 2),
    }
🚨 400 invalid_request_error — messages.2.content.0.tool_result.content: Found an object, but `tool_result` content must either be a string or a list of content blocks.
The beta runner does not serialize a returned dict for you. Every @beta_tool function in this lab must end with return json.dumps(result) — a plain str — exactly like the manual loop's "content": json.dumps(result) line.
